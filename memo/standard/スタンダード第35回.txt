REST API


REST : 設計原則
API : プログラム


Webブラウザ　→(HTTPリクエスト)→　REST API(サーバー)　→(レスポンス)→　Webブラウザ　
・HTTPリクエスト（GET/POST/PUT/DELETE）
・レスポンス（JSON,XML）


REST APIの４つの設計原則

1ステートレス性
2統一インターフェイス
3アドレス可能性
4接続性


1ステートレス性
サーバーやシステムがリクエストごとに状態（state）を保持しない性質
・各リクエストが「誰から来たか」覚えていない
・毎回リクエスト時に必要な情報(IDなど)を含めて送らなければならない 例）毎度ログインしなければならない

2統一インターフェイス
HTTPメソッド（GET/POST/PUT/DELETE）

3アドレス可能性
URL

4接続性
URLでリソース同士を繋げる


RESTの原則
1リソース指向アーキテクチャ
2ステートレス性
3キャッシュ
4統一インターフェイス
5クライアント・サーバー


1リソース指向アーキテクチャ
リソース1つに対してURLが1つだけ存在する

2ステートレス性
※ステートフル　：　システムが常態を保持し続ける　例）ずっとログインしたまま

3キャッシュ
２回目以降(再度)同じ情報を素早く取得する

4統一インターフェイス
HTTPメソッド（GET/POST/PUT/DELETE）

5クライアント・サーバー
クライアント　：　Webブラウザ
サーバー　：　Webシステム


GET/api/users/{user_id}


REST APIではリクエストとレスポンスはJSON形式でデータを送る


ヘッダーにトークンを持たせて毎回ログイン


オーバーフェッチとアンダーフェッチによりサーバーやネットワークに負荷がかかる
オーバーフェッチ：クライアントが必要以上に情報を取得する
アンダーフェッチ：必要な情報を取得するために複数回のリクエストを行う
